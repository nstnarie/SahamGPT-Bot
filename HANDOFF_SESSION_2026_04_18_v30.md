Removed all synthetic foreign flow from the system. Real Asing broker data (2.6M rows, 137 tickers, 2024-2025) is now the only source of foreign flow signals. Solved the GitHub 100MB file size limit problem with a pre-aggregation approach.

### Problem Solved: GitHub 100MB File Size Limit
Raw `broker_summary` = 2.6M rows = ~450MB total. Can't commit. Solution: pre-aggregate into `foreign_flow` (one row per ticker/date instead of ~18 broker rows per ticker/date). The lean `idx_broker_part_a.db` is only 4.8MB containing 72,167 `foreign_flow` rows.

### Aggregation SQL Used
```sql
DELETE FROM foreign_flow WHERE date >= '2024-01-02' AND date <= '2025-12-30';

INSERT OR IGNORE INTO foreign_flow
    (ticker, date, net_foreign_value, net_foreign_volume,
     foreign_buy_value, foreign_sell_value)
SELECT
    ticker, date,
    SUM(CASE WHEN broker_type = 'Asing' THEN net_value ELSE 0 END),
    SUM(CASE WHEN broker_type = 'Asing' THEN net_volume ELSE 0 END),
    SUM(CASE WHEN broker_type = 'Asing' AND net_value > 0 THEN net_value ELSE 0 END),
    SUM(CASE WHEN broker_type = 'Asing' AND net_value < 0 THEN ABS(net_value) ELSE 0 END)
FROM broker_summary
GROUP BY ticker, date;
-- Inserted 62,988 real rows
```

## Backtest Results with Real Asing Data

### 2024 (real data, MPW=6)
- **PF 0.53, -13.7%, 101 trades, 34% WR** — still losing
- Worse than synthetic (PF 1.40/+6.4%)

### 2025 (real data, MPW=6)
- **PF 1.65, +19.7%, 98 trades, 46% WR** — profitable

### Root Cause of 2024 Degradation
Real Asing flows in 2024 are LESS NEGATIVE than synthetic estimated. This means:
- KSEI filter (`ksei_net_5d < -5B`) is less restrictive with real data
- More trades get through in 2024, including more bad ones
- TREND_EXIT degraded: 12 trades/+168M (synthetic) → 9 trades/+40M (real)
- Emergency stops increased: 8 (synthetic) → 16 (real)

The MPW=6 fix worked on synthetic because synthetic was accidentally acting as a quality pre-filter. Real data removes this accidental filter, exposing the underlying 2024 regime problem.

## File: `idx_broker_part_a.db` Schema
```sql
CREATE TABLE foreign_flow (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    ticker TEXT NOT NULL,
    date TEXT NOT NULL,
    net_foreign_value REAL,   -- positive = net buy, negative = net sell
    net_foreign_volume REAL,
    foreign_buy_value REAL,   -- sum of positive net_value rows
    foreign_sell_value REAL,  -- sum of ABS(negative net_value rows)
    UNIQUE(ticker, date)
);
```
